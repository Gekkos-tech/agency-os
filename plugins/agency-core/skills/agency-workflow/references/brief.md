# Phase 1 — BRIEF

Goal: a written brief the client would sign. Everything later traces back to it.

## Intake interview

Ask only what the request didn't already answer. Core set:

1. **Business** — what does the client do, for whom, in which market/city?
2. **Goal** — the ONE thing the site must achieve (bookings, leads, sales,
   credibility). If the client names three, force-rank them.
3. **Audience** — who visits, on what device, with what intent?
4. **Scope** — pages/features (reservation, shop, blog, multilingual?).
5. **Content** — what exists (texts, photos, logo, brand colors)? What must
   be produced?
6. **Constraints** — budget band, deadline, tech requirements (CMS, hosting,
   existing stack), legal context (for DACH clients: DSGVO, Impressum, BFSG).
7. **Taste anchors** — 2–3 sites the client likes and why; competitors.

For a compressed mini-brief (when the user wants to move fast), answer 1–4
yourself from context, state assumptions explicitly, and flag them for
confirmation.

## Brief document template

```markdown
# Project Brief — {client}
## Business & goal
## Audience & top user tasks
## Scope (pages / features)          ← in-scope AND explicitly out-of-scope
## Content inventory                 ← have / need / who delivers
## Constraints                       ← budget band, deadline, tech, legal
## Taste anchors & competitors
## Success criteria                  ← measurable, e.g. "reservation in ≤ 3 taps"
## Open questions
```

Write it in the client's language. Save as `docs/brief.md` in the project.

## Exit gate

- [ ] One primary goal, force-ranked if several
- [ ] Scope has an explicit out-of-scope list
- [ ] Content inventory says who delivers what by when
- [ ] Every assumption is marked as assumption
- [ ] Client (or user) confirmed the brief — or explicitly waived confirmation
